# FitAI Coach ‚Äî Project Documentation

## 1. Project Overview

**FitAI Coach** is a Python-based desktop application that delivers personalized nutrition and workout plans through two independent machine learning pipelines. The system trains and compares five classification algorithms, selects the best-performing model per task, and surfaces predictions through a polished CustomTkinter GUI.

The project comprises three layers:

| Layer | Files | Purpose |
| :--- | :--- | :--- |
| **Data** | `diet_recommendations_dataset.csv`, `workout_dataset.csv` | Training data for the two ML pipelines |
| **ML Core** | `health_app.py`, `benchmark.py`, `workout_benchmark.py`, `dataset_generation.py` | Model training, evaluation, and benchmarking |
| **GUI** | `main.py`, `gui/` | User interface and view routing |

## 2. Datasets

### Diet Dataset ‚Äî Kaggle (Real-World)
`diet_recommendations_dataset.csv` ‚Äî 1,000 records sourced from Kaggle.

| Column | Type | Notes |
| :--- | :--- | :--- |
| Age, Weight\_kg, Height\_cm, BMI | Numeric | Physical attributes |
| Gender, Disease\_Type, Severity | Categorical | Encoded for ML |
| Physical\_Activity\_Level | Categorical | Sedentary / Moderate / Active |
| Cholesterol\_mg/dL, Blood\_Pressure\_mmHg, Glucose\_mg/dL | Numeric | Clinical measurements |
| Weekly\_Exercise\_Hours | Numeric | Self-reported lifestyle metric |
| Diet\_Recommendation | **Target** | Balanced / Low\_Carb / Low\_Sodium |

`Dietary_Restrictions` is excluded from features to prevent data leakage (values like "Low\_Sodium" would directly reveal the target label).

`Disease_Type` contains 204 `NaN` rows (healthy patients) ‚Äî filled with `'None'` before encoding.

### Workout Dataset ‚Äî Synthetic
`workout_dataset.csv` ‚Äî 1,000 records generated by `dataset_generation.py`.

Labels (`Workout_Intensity`: Light / Moderate / Intense) are assigned by deterministic clinical rules (disease, BMI, age, activity, goal) with **8% random label noise** injected deliberately. This noise prevents perfect determinism, ensures genuine variance between classifiers, and simulates real-world measurement uncertainty.

## 3. Machine Learning Pipelines

### 3.1 Diet Recommender

**Class**: `DietRecommenderAI` in `health_app.py`

**Features (12)**:
`Age`, `Gender`, `Weight_kg`, `Height_cm`, `BMI`, `Disease_Type`, `Severity`, `Physical_Activity_Level`, `Cholesterol_mg/dL`, `Blood_Pressure_mmHg`, `Glucose_mg/dL`, `Weekly_Exercise_Hours`

**Target**: Diet type ‚Äî `Balanced`, `Low_Carb`, `Low_Sodium`

**Training**: Full dataset (1,000 rows) with `LabelEncoder` on categorical columns. Trains a `RandomForestClassifier` (100 estimators) and stores the fitted encoders to replicate the same transformation at prediction time.

**Inference in the app**: Clinical measurements (Cholesterol, Blood Pressure, Glucose) are not collected from the user. Instead, disease-conditioned population averages are used internally:

| Medical Condition | Cholesterol (mg/dL) | Blood Pressure (mmHg) | Glucose (mg/dL) |
| :--- | :---: | :---: | :---: |
| None | 170 | 115 | 85 |
| Diabetes | 190 | 125 | 160 |
| Hypertension | 205 | 150 | 95 |
| Obesity | 215 | 135 | 108 |

This design choice removes the burden of clinical knowledge from the end-user while still supplying the model with statistically representative inputs for each condition.

**Output**: `predict_with_confidence()` returns a `(diet_type: str, confidence: float)` tuple. Confidence is the maximum class probability from `predict_proba()`.

### 3.2 Workout Intensity Recommender

**Class**: `WorkoutRecommenderAI` in `health_app.py`

**Features (8)**:
`Age`, `Gender`, `Weight_kg`, `Height_cm`, `BMI`, `Disease_Type`, `Physical_Activity_Level`, `Goal`

**Target**: Workout intensity ‚Äî `Light`, `Moderate`, `Intense`

**Training**: Full synthetic workout dataset with the same RF + LabelEncoder approach. Activity level and goal labels are mapped from GUI-friendly values ("Moderately Active" ‚Üí "Moderate", "Maintain Weight" ‚Üí "Maintain") before encoding.

**Output**: `predict()` returns `(intensity: str, confidence: float)`.

## 4. Model Benchmarking

Both pipelines are evaluated in dedicated benchmark scripts before selecting Random Forest as the production model.

### Evaluation Protocol

- **Split**: 80% train / 20% test (stratified)
- **Cross-validation**: 5-fold stratified CV on the training set (F1 weighted)
- **Metrics**: Accuracy, Precision, Recall, F1, Average prediction confidence

### Models Compared

| Model | Library Class |
| :--- | :--- |
| Logistic Regression | `LogisticRegression(max_iter=1000)` |
| Decision Tree | `DecisionTreeClassifier` |
| Random Forest | `RandomForestClassifier(n_estimators=100)` |
| K-Nearest Neighbors | `KNeighborsClassifier(n_neighbors=7)` |
| Gradient Boosting | `GradientBoostingClassifier(n_estimators=100)` |

### Diet Benchmark (`benchmark.py`)
The diet dataset is highly structured (disease type strongly determines diet category), so tree-based models and Logistic Regression converge near-perfect accuracy. KNN, which is distance-based and sensitive to unscaled features, shows significantly lower performance ‚Äî demonstrating that algorithm choice matters even on well-structured data.

### Workout Benchmark (`workout_benchmark.py`)
The 8% label noise in the synthetic workout dataset produces meaningful model differentiation:
- Random Forest and Gradient Boosting: ~83‚Äì84% F1
- Decision Tree: ~82% F1
- Logistic Regression: ~64% F1
- KNN: ~61% F1

## 5. Mathematical Foundations

### BMR ‚Äî Mifflin-St Jeor Equation
$$BMR_{male} = 88.362 + (13.397 \times w) + (4.799 \times h) - (5.677 \times a)$$
$$BMR_{female} = 447.593 + (9.247 \times w) + (3.098 \times h) - (4.330 \times a)$$

### TDEE (Total Daily Energy Expenditure)
$$TDEE = BMR \times ActivityMultiplier$$

| Activity Level | Multiplier |
| :--- | :---: |
| Sedentary | 1.200 |
| Lightly Active | 1.375 |
| Moderately Active | 1.550 |
| Very Active | 1.725 |
| Extra Active | 1.900 |

### Caloric Target
- **Lose Weight**: TDEE ‚àí 500 kcal/day (‚âà 0.45 kg/week deficit)
- **Maintain Weight**: TDEE
- **Gain Weight**: TDEE + 500 kcal/day

### Macronutrient Ratios

| Diet Type | Protein | Fats | Carbs |
| :--- | :---: | :---: | :---: |
| Balanced | 30% | 30% | 40% |
| Low\_Carb | 30% | 50% | 20% |
| Low\_Sodium | 30% | 30% | 40% |

Grams are derived from: Protein & Carbs = 4 kcal/g, Fats = 9 kcal/g.

## 6. Desktop GUI Application

### Architecture ‚Äî View-Router Pattern

```
main.py (App root / router)
‚îú‚îÄ‚îÄ SetupView       (data entry ‚Üí triggers finish_setup())
‚îî‚îÄ‚îÄ [after setup]
    ‚îú‚îÄ‚îÄ Sidebar     (navigation)
    ‚îî‚îÄ‚îÄ Content frame
        ‚îú‚îÄ‚îÄ DashboardView
        ‚îú‚îÄ‚îÄ DietView
        ‚îî‚îÄ‚îÄ WorkoutView
```

`main.py` acts as a state machine. It starts with `SetupView` in a `1√ó1` grid. When the user submits the form, `finish_setup()` destroys the setup frame, reconfigures the root to a sidebar + content layout, and passes the computed `user_data` dict to all views.

### SetupView ‚Äî User Input Form

Collects only information a non-clinical user can self-report:

| Field | Type |
| :--- | :--- |
| Name | Text |
| Age | Integer |
| Gender | Dropdown (Male / Female) |
| Weight (kg), Height (cm) | Decimal |
| Medical Condition | Dropdown (None / Hypertension / Diabetes / Obesity) |
| Severity | Dropdown (Mild / Moderate / Severe) |
| Weekly Exercise (hrs) | Decimal |
| Activity Level | Dropdown (5 levels) |
| Primary Goal | Dropdown (Lose / Maintain / Gain Weight) |

Clinical measurements (Cholesterol, Blood Pressure, Glucose) are not shown to the user ‚Äî they are inferred from the selected medical condition using population-average defaults (see Section 3.1).

On submission, `SetupView` calculates BMR/TDEE, queries both ML models, and stores a `user_data` dict with all results for the views to consume.

### DashboardView

Displays a high-level summary:
- AI recommendation pills (diet type + workout intensity with confidence %)
- BMI gauge (240¬∞ polar arc, 15‚Äì40 scale)
- Macronutrient donut chart
- Calorie progress bar (target vs. consumed)
- Meal plan cards (Breakfast 25% / Lunch 35% / Dinner 30% / Snack 10%)

### DietView

Populated from the `DIET_INFO` dictionary in `health_app.py`:
- Diet name, description, and clinical rationale
- Macro progress bars (grams + % of calories)
- Calorie split across four meals
- Three-column guidelines grid: recommended foods / foods to avoid / practical tips
- Expanded meal cards with calorie targets per meal

### WorkoutView

Driven by the `WORKOUT_PLANS` dictionary in `health_app.py`:
- Intensity badge (color-coded: green / amber / red)
- Duration and frequency stat boxes
- Four exercise cards with left accent stripe and detail text
- Model confidence percentage in the header bar

### Component System (`gui/components/`)

Custom Matplotlib charts embedded via `FigureCanvasTkAgg`:
- **BMI Gauge**: 240¬∞ polar plot with color-coded zones and a needle cap
- **Donut Chart**: Trigonometric wedge layout with manually offset labels to prevent overlap
